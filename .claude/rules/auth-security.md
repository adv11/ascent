# Auth, CSP/SRI, and security-critical ordering

Relocated from `CLAUDE.md` (issue #86) with no content changes — see
`docs/adr/ADR-007-agent-memory-architecture.md`. Several of these are ordering
requirements (delete data before deleting the Auth record, etc.) where getting the
sequence backwards silently orphans data or blocks cleanup — read carefully before
touching auth flows.

**SRI + CSP — mandatory when loading CDN scripts.** `index.html` locks the three Firebase
SDK modules with `<link rel="modulepreload" integrity="sha384-...">` Subresource Integrity
hashes. When upgrading the Firebase SDK version, four things must change in lockstep:
(1) all three import URLs in `src/services/firebase.js`, (2) all three `href` attributes
in the `<link rel="modulepreload">` tags, (3) all three `integrity` attributes in those
same tags, (4) the hash table in `docs/adr/ADR-002-csp-sri-security.md`. Missing any one
of these will cause a hash mismatch and the app will fail to boot. Regenerate hashes with:
`curl -s <url> | openssl dgst -sha384 -binary | base64`. The Content Security Policy
in `index.html` must stay consistent with any new CDN domains added; update both the CSP
meta tag and the `firebase.json` hosting headers.

**Password utility module — `src/ui/utils/password.js`.** Two shared exports used by both auth pages:
- `scorePassword(s)` — pure function returning 0–4. 0 means empty or < 6 chars; 1 = base (≥6 chars), +1 each for ≥8 chars, ≥12 chars, mixed case, digit, special character (capped at 4). No external library — do not replace with a dependency.
- `makePasswordToggle(input)` — factory that creates an absolutely-positioned `<button class="password-toggle">` that toggles `input.type` between `'password'` and `'text'` and updates `aria-label` to "Show password" / "Hide password". Caller must wrap the input in a `<div class="field-input-wrap">` (CSS `position: relative; display: grid;`) — the `.field-input` inside automatically gets `padding-right: 52px` via the cascade.

**Password reset uses Firebase's default action URL (Option A).** `authApi.sendResetEmail(email)` calls `sendPasswordResetEmail(auth, email)` and relies on Firebase's hosted action page for the "Set new password" form — no custom reset-confirm route exists in-app. Do not implement a custom reset form (`#/reset-password?oobCode=...`) unless explicitly requested. The sign-in page manages the reset request inline (same card, no route navigation) by swapping `bodySlot` content and updating `titleEl`/`subtitleEl` from the `authShell` return value. The success state deliberately shows the same UI regardless of whether the email belongs to an existing account — this prevents account-existence enumeration.

**Account deletion must delete `users/{uid}` from Realtime Database before calling `deleteUser()`.** Reversing this order leaves orphaned data in the database — once the Auth record is gone, the security rules block cleanup because there is no longer an authenticated user. `authApi.deleteAccount(password)` re-authenticates first (`reauthenticateWithCredential`) to obtain a fresh token before the deletion. If Firebase throws `auth/requires-recent-login`, surface the error message from `authErrorMessage` (already mapped) rather than attempting deletion. Never auto-delete without explicit password confirmation from the user.

**Anonymous Firebase Auth users must be explicitly deleted when they exit without linking, to prevent orphaned data (issue #24).** A guest session's anonymous UID can never be re-authenticated once the session ends, so if it's never linked to a real account (`authApi.linkGuest`), its roadmap data would otherwise sit in the database forever with no way to read, export, or delete it. `authApi.signOut()` (`src/services/firebase.js`) delegates to `signOutWithCleanup()` (`src/services/authCleanup.js` — a pure, dependency-injected function with no Firebase imports, kept separate specifically so it's unit-testable without a real Firebase project) which checks `user.isAnonymous` first: for an unlinked guest it removes `users/{uid}` from the database and then deletes the Auth record (same data-before-Auth-record order as `deleteAccount()`, and for the same reason), instead of a plain sign-out. A linked guest is no longer anonymous by the time they sign out, so that path is unaffected. If the cleanup call throws (e.g. a stale token), it falls back to a plain sign-out — never block the user from leaving the app over a cleanup failure. See `docs/adr/ADR-005-anonymous-user-lifecycle.md`. `signOutWithCleanup()` also triggers `roadmapStore.js`'s and `dailyTodoStore.js`'s sign-out contract (see `.claude/rules/roadmap-store.md`) — the uid transition clears their local data too.

**Every sign-out entry point must go through `confirmAndSignOut()` (`src/ui/utils/signOut.js`), never call `authApi.signOut()` directly.** Previously `sidebar.js` had its own inline sign-out handler that only confirmed for an anonymous guest with unsaved (`dirty`) changes — a real account, or a guest with nothing unsaved, signed out instantly with zero confirmation, which is surprising (and for a guest, irreversible — their local-only data is gone the moment they confirm). `confirmAndSignOut(user, store)` always shows a `confirmDialog()` first, with the message tailored to what's actually at stake (a plain "you'll need to sign in again" for a real account; a data-loss warning, `danger: true`, for a dirty guest). Both current call sites — `sidebar.js`'s footer button (dashboard/app-shell pages) and `onboarding.js`'s top-row button (added because that page had no sign-out affordance at all before) — use this one shared function so behavior never drifts between them. If you add another sign-out entry point, import and call `confirmAndSignOut`, don't reimplement it.

**`connect-src` needs `'self'` for same-origin JSON module imports, not just third-party API hosts (issue #20).** A JS `import ... with { type: 'json' }` is fetched by the browser under CSP's `connect-src`, not `script-src` — `index.html`'s `connect-src` originally listed only Firebase/local-emulator origins, which silently broke every page's module graph (including sign-in) the moment `src/data/changelog.js` imported `changelog.json` this way, invisible to `npm test`/`npm run lint` since neither enforces CSP. `'self'` is now in `connect-src` for exactly this reason. If you add another same-origin `fetch`/XHR/JSON-module-import, it's already covered; don't remove `'self'` without re-checking every same-origin network call in the app.

**Guest data-loss messaging (issue #123).** An anonymous guest's roadmap lives only in `localStorage`/that anonymous Firebase Auth identity until linked to a real account (`authApi.linkGuest`, used by `signUp.js` when `auth.currentUser?.isAnonymous`) — nothing warned about that risk before this. `sidebar.js`'s identity area shows a persistent `.app-sidebar-guest-risk` icon (tooltip via `attachTooltip`) whenever `user.isAnonymous`, and `dashboard.js` calls `maybeShowGuestDataRiskNudge({ user, store })` (`src/ui/components/guestDataRiskNudge.js`) once per render — a `confirmDialog` (not a new UI primitive) offering to navigate to `/signup`, shown at most once ever per uid, gated behind `shouldShowGuestRiskNudge()`/`markGuestRiskNudgeShown()` (`src/ui/utils/guestDataRisk.js`) once at least `COMPLETED_THRESHOLD` (5) items are done — the same `guestRiskNudgeShownKey(uid)` per-uid, device-level `localStorage` pattern as `backupFirstSeenAtKey`/`lastBackupAtKey` above. Both the indicator and the nudge are strictly informational — they don't touch `linkGuest`/`signOutWithCleanup`/any anonymous-user lifecycle mechanics.

**Realtime Database rules structural constraint.** `firebase/database.rules.json` has a `$other: { ".validate": "false" }` catch-all under `users/$uid` specifically to stop a buggy or malicious client from writing arbitrary data outside the known shape (`roadmap`/`roadmaps`/`meta`/`dailyTodos`/`activityLog`/`reports`) — still auth-scoped to that uid, just unbounded before this. If you add a genuinely new top-level field under a user's data, add an explicit `.validate` rule for it here — never rely on the `$other` catch-all rejecting it silently as "good enough." **Every subtree that has its own named children needs its own `$other: false` catch-all, not just the top-level `users/$uid` one** — a catch-all one level up does not reach down into a nested object's own unvalidated fields (issue #275 found three subtrees missing this: the legacy `users/$uid/roadmap/items/$itemId`, `dailyTodos/$todoId`, and `streakFreezes` — all fixed to match the pattern already established at `roadmaps/$templateId`, `meta`, and `sharedRoadmaps/$shareId/items/$itemId`). When you add a new object-shaped subtree (anything with named children of its own, not a single scalar/boolean field), add its `$other: false` catch-all in the same PR that introduces it — don't leave it for a later audit to catch. The client-side enforcement half (item/todo count caps, title/resource length caps) lives in `roadmapStore.js`/`dailyTodoStore.js` — see `.claude/rules/roadmap-store.md`. In-app feedback reports (issue #9) get a top-level `reports/{reportId}` write-only path plus a `users/{uid}/reports/{reportId}` personal-history summary — see `.claude/rules/roadmap-store.md`'s feedback section for the full model.

**Server-side data caps close the gap client-side-only enforcement leaves open (issue #122).** Until this issue, almost every length/shape cap this app enforces (`src/core/roadmap/limits.js`'s `MAX_TITLE_LENGTH`/resource label/url caps, `feedbackRateLimit.js`'s rate limit) only ran in client-side JavaScript — any authenticated user (guest or real account) could bypass the app entirely and write directly to the Realtime Database REST API against their own uid-scoped path with none of it applied. Not a cross-user confidentiality leak (every path was already correctly uid-scoped), but a real storage-cost/DoS/data-integrity gap. `firebase/database.rules.json` now adds `.validate` rules mirroring the client-side caps wherever Realtime Database rules can actually express them: `users/{uid}/roadmap(s)/{templateId}/items/{itemId}.title` (≤200, matching `MAX_TITLE_LENGTH`), `.notes` (≤5000, matching `itemPanel.js`'s `NOTES_MAX_LENGTH`), and `.resources.{index}.label`/`.url` (≤120/≤2048, matching `MAX_RESOURCE_LABEL_LENGTH`/`MAX_RESOURCE_URL_LENGTH`); `meta.customRoadmaps.{index}.title`/`.description` (≤200/≤1000, new `MAX_CUSTOM_ROADMAP_TITLE_LENGTH`/`MAX_CUSTOM_ROADMAP_DESCRIPTION_LENGTH` in `limits.js`) plus an array-size cap via the same index-whitelist idiom `favoriteRoadmapIds` (issue #177) already established — `$index.matches(/^[0-9]{1,3}$/)`, capping at 1000 entries, since `numChildren()` still isn't reliably supported; `activityLog.$date` now requires the key itself to match `/^[0-9]{4}-[0-9]{2}-[0-9]{2}$/` (not just any string) and the value to be `0`-`5000`; and the top-level `reports/{reportId}.screenshotB64` is capped at 700,000 characters — `screenshotCapture.js`'s `MAX_SCREENSHOT_BYTES` (500KB) inflated by base64 encoding (~4/3×) plus headroom, not a bare byte-for-byte translation. `roadmapStore.js`'s `createCustomRoadmap()` clamps (`.slice()`, not reject) an oversized title/description to these same caps before ever writing, so a legitimate call can never be rejected by its own server rule. **Two caps remain deliberately client-side only, not an oversight**: a per-roadmap item **count** (`MAX_ITEMS_PER_ROADMAP`, `roadmapStore.js`'s `addItem()`) — Realtime Database rules genuinely cannot count a map's children — and feedback-report rate limiting (`feedbackRateLimit.js`) — RTDB rules can't express "reject if this uid wrote N times in the last M seconds"; either would need a Cloud Function, which this static-hosted app doesn't have. Every cap above was verified against a real local Firebase emulator, not just assumed to parse — see `tests/e2e/dataCapRules.test.js` (same "a `.validate` rule that fails to parse silently serves the emulator's permissive default for the whole namespace, not just rejects the one path" caution `favoriteRoadmapIds`' own rule already documents in `.claude/rules/roadmap-store.md`).

**Issue #187 closed a gap adjacent to but distinct from #122's own scope: `roadmaps/{templateId}/items` had no per-field schema beyond the length caps above, `phases` had zero validation at all, and neither `roadmaps/{templateId}` nor `meta` rejected an undeclared top-level key.** `#122` capped `title`/`notes`/resource `label`/`url`, but left the rest of an item's shape wide open — a client could set `priority` to any string, `done`/`custom`/`deleted` to any value, or grow `resources` unbounded. `firebase/database.rules.json`'s `users/{uid}/roadmaps/{templateId}/items/{itemId}` now also validates `priority` (must be one of `P0`/`P1`/`P2`/`P3`, the same enum `roadmapStore.js`/`importValidator.js`/`schemaAdapter.js` already use), `phase`/`section` (≤200, matching title), `done`/`custom`/`deleted` (booleans), `createdAt` (a number), and caps `resources` to indices `0`-`99` (the same index-whitelist idiom as `favoriteRoadmapIds`/`meta.customRoadmaps`, capping at 100 resources per item) — `title` also now rejects empty, not just oversized. `phases` (a custom roadmap's own phase/section structure, `.claude/rules/roadmap-store.md`'s custom-roadmap-ids section — an array of `{ id, title, priority, resourceKey, sections: [{ id, title }] }`, persisted alongside `items` on every `saveRoadmap()`) previously had no rule whatsoever; it's now capped at indices `0`-`999`, each phase requiring a non-empty ≤200-char `title`, an optional `P0`-`P3` `priority`, and up to `0`-`999` `sections`, each requiring its own non-empty ≤200-char `title`. Both `roadmaps/{templateId}` and `meta` gained a `$other: { ".validate": "false" }` catch-all matching the existing pattern at `users/{uid}` itself, so a client can no longer write an arbitrary extra key under either path. **`sharedRoadmaps` was deliberately left untouched** — tracked separately by issue #188 to avoid a merge conflict between the two issues' parallel work on the same file. Verified against a real local Firebase emulator, including that `tests/e2e/customRoadmap.test.js`/`importRoadmap.test.js`/`customRoadmapRace.test.js`'s full existing suite still passes unmodified under the new rules — see the two new cases added to `tests/e2e/dataCapRules.test.js`.

**`sharedRoadmaps/{shareId}` is a deliberate, narrowly-scoped exception to "every rule is uid-scoped" — issue #131.** Every rule above this one in `firebase/database.rules.json` reads as `auth != null && auth.uid == $uid` (or a narrower same-uid check); `sharedRoadmaps/{shareId}` is the one path with `".read": true` — an intentional public, unauthenticated read, not an oversight to flag in a future security pass. It's a **top-level** path (not nested under `users/{uid}`) specifically so no rule ever needs to let one uid read another's `users/{uid}/...` subtree — the read-only sharing feature never widens that constraint at all. Writes stay tightly scoped: `".write"` allows either a **create** (`!data.exists()`, requiring `newData.child('ownerUid').val() == auth.uid` so a client can't forge a share under someone else's identity) or a **full delete by the original owner** (`data.exists() && data.child('ownerUid').val() == auth.uid && newData.val() === null`) — there is no allowed path for editing an already-published snapshot's content, by anyone, including the original publisher; republishing always mints a fresh `shareId` instead (see `.claude/rules/roadmap-store.md`'s "Roadmap sharing" section). The owner's own share-id index lives at `users/{uid}/meta/shareIds` (a plain string array, same shape/validation as `hiddenTemplateIds`), which stays under the normal uid-scoped rule tree above — only the snapshot itself is public.

**`sharedRoadmaps/{shareId}/items` is now validated to the exact narrow shape `shareSchema.js`'s `toShareItem()` actually emits — issue #188.** Before this, `items` wasn't even in the top-level `hasChildren([...])` required-field list and had zero validation underneath — the one publicly-readable path in the app had no cap on the size, count, or shape of the one field most likely to be large. `items` is now required (a share snapshot always includes it, per `buildRoadmapShareSnapshot()`), and each `$itemId` must `hasChildren(['title', 'phase', 'section', 'priority', 'done'])` with `title`/`phase`/`section` capped at 200 chars (matching `MAX_TITLE_LENGTH`/`MAX_SHARE_TITLE_LENGTH`'s existing precedent), `priority` capped at 50, `done` a boolean, an optional `resources.$index` capped the same as every other resource list in this file (`label` ≤120, `url` ≤2048, index `$index.matches(/^[0-9]{1,3}$/)`), and a trailing `"$other": { ".validate": "false" }` that rejects any field outside this list — `notes`/`completedAt`/tags or anything else `toShareItem()` deliberately never emits can't be smuggled onto the public path even if a client bypasses `shareStore.js` entirely. One limitation carried over from issue #122's own note: RTDB rules still can't cap the total **number** of items in the map (no `numChildren()` support for this), so an authenticated user can still publish a share with an unreasonably large number of individually-compliant items — same accepted gap as the private `roadmap(s)/{templateId}/items` count, not new here. Verified against the Firebase emulator in `tests/e2e/dataCapRules.test.js`'s `sharedRoadmaps/{shareId}/items` case (oversized title, missing required field, a field outside the schema, oversized resource url — all rejected; a compliant item accepted) — see that file and `roadmapSharingRules.test.js` for the ownership/read/revoke coverage this addition doesn't duplicate.

**`users/{uid}/reports/{reportId}` now mirrors the top-level `reports/{reportId}` rule's `type`/`title` validation — issue #192.** `submitReport()` writes the same report to both paths in one multi-path update, but the per-user history mirror's `.validate` previously only checked `hasChildren(['type','title','submittedAt'])` — no `type` enum, no `title` length cap — while the top-level path already enforced `type.matches(/^(bug|feature|feedback)$/)` and `title.length <= 120`. Not a cross-user leak (the outer `auth.uid == $uid` write rule already scopes this path to its own owner), but a real shape inconsistency for what's logically the same write: a client bypassing `feedbackStore.js`/`feedbackRateLimit.js` entirely could write an invalid `type` or an arbitrarily long `title` into their own report history. The per-user copy now requires the identical `type`/`title` shape as the top-level rule (it deliberately doesn't re-check `userId`, since that field isn't part of this path's schema and the uid-scoping is already structural). Verified against the Firebase emulator in `tests/e2e/dataCapRules.test.js`.
