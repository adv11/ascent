# Changelog

All notable changes to this project will be documented in this file.

## [Unreleased]

### Added
- **Multi-roadmap support — concurrent progress per template, non-destructive switching** (issue #58): previously the app had exactly one roadmap slot per account (`users/{uid}/roadmap`), and switching starter templates always replaced it wholesale — a user tracking Frontend for a job search and Piano as a hobby at the same time couldn't keep both moving, since picking one destroyed the other. Each started template now gets its own Firebase node, `users/{uid}/roadmaps/{templateId}/`, tracked in `users/{uid}/meta.startedTemplateIds` with `meta.activeTemplateId` naming the one currently displayed. `roadmapStore.js`'s `switchRoadmap(templateId)` replaces `initFromTemplate()` — an already-started template loads its own saved progress (cache-first: in-memory this session, then Firebase, then local storage, then seed), a not-yet-started one seeds fresh, and neither ever touches another template's data. The onboarding picker (`src/ui/pages/onboarding.js`) drops its "this replaces your current roadmap" confirmation entirely — switching is now always instant and non-destructive — and gains a third card badge state, "In progress", for a template that's been started but isn't currently active; a started template that's also hidden stays visible (badged) instead of disappearing, since hiding only ever filters the "start something new" grid. Locally, `KEYS.ROADMAPS` (`ascent-roadmaps-v1`) replaces the single `KEYS.ROADMAP` blob with a per-template keyed shape; existing local data is migrated forward automatically, leaving the old key in place. Existing (pre-#58) Firebase accounts are migrated transparently on their next sign-in — the old singular `users/{uid}/roadmap` path is copied into the new per-template path and left in place as a safety net, never deleted. `firebase/database.rules.json` extended with the `roadmaps/$templateId` node and `meta.activeTemplateId`/`meta.startedTemplateIds` validators, alongside (not replacing) the legacy validators. Three correctness fixes specific to per-template Firebase paths: a debounced edit pending on the template you're switching *away from* is now flushed to its own path before the switch, instead of risking being silently dropped once the timer later fires against the newly active template; a closure-based "stale listener guard" (each template's realtime listener is scoped to the exact template it was attached for) replaces the old #51 cross-template echo payload tag, which is no longer needed now that each template has its own path; and (found via real E2E testing against live Firebase, not by the mocked test suite) the realtime listener now never applies an incoming snapshot while a local edit is still unflushed (`dirty === true`) — a queued-or-in-flight edit is always newer than anything the listener could be echoing, and Firebase's own data normalization means a delayed echo of an older write doesn't always byte-for-byte match what was sent, so string-matching alone could occasionally misclassify a stale echo as "genuinely newer" and silently revert an edit made in the narrow window right after starting a fresh (not-yet-started) template.
- **Starter template system + new-user onboarding picker** (issue #51): the app is no longer hardwired to the Java Backend Engineer roadmap for every new sign-up. `src/data/templates/` is a new template registry with four starter templates — `java-backend.js` (the original roadmap, moved verbatim, unchanged), `frontend.js` (HTML/CSS/JavaScript/TypeScript/React/testing/tooling/performance/accessibility/system design), `data-science.js` (Python/math/statistics/ML/deep learning/NLP/SQL/data engineering/MLOps), and `blank.js` (four empty phases — Learn, Practice, Build, Review — with no seeded topics). `src/data/roadmap.js` is now a thin backward-compat re-export shim pointing at `templates/java-backend.js` — no logic lives there anymore. A new `/onboarding` route (`src/ui/pages/onboarding.js`) shows a template picker to any signed-in user who hasn't chosen a template yet; picking a card calls the new `roadmapStore.initFromTemplate(templateId)` and lands on `/app`. `roadmapStore.js`'s `setUser` is now async: on every sign-in it does a one-time read of `users/{uid}/meta` and `users/{uid}/roadmap` from Firebase to decide whether this user still needs onboarding, treating any account with real remote or local progress (or an explicit `meta.onboardingDone` flag) as already onboarded — this backfills the flag for every pre-existing account with no forced migration step. `dashboard.js` now groups items using the current template's phase/section skeleton (`store.getSnapshot().phases`) instead of a hardcoded import, so a template like "blank" can render empty phase-cards with zero items. Firebase Database Rules extended to validate `users/{uid}/meta/templateId` (string) and `meta/onboardingDone` (boolean).
- Also fixed the last remaining Java-specific string outside the templates themselves: the dashboard search input's placeholder ("Search topics, e.g. Kafka, RAG, Saga…" → "Search topics…").
- **"Switch template" from the dashboard** (issue #51 follow-up): the onboarding picker is no longer a strict one-way gate — a new "Switch template" link in the dashboard header lets a user return to `/onboarding` at any time. Reached this way, the page shows a "← Back to my roadmap" link and requires a confirmation before replacing the current roadmap, since picking a new template discards the current one. First-time onboarding (right after sign-up) is unaffected — no back link is shown and no confirmation is required there, since there is nothing to lose yet.
- **Much deeper Frontend Developer and Data Scientist templates** (issue #51 follow-up): both templates were noticeably thinner than the Java Backend roadmap. `frontend.js` grew from 116 to 230 topics across 14 phases (added TypeScript advanced types, deeper React (hooks, state management, SSR), styling/design systems, testing, tooling, performance, accessibility, security, deployment/DevOps, PWA, and frontend system design), and `data-science.js` grew from 80 to 166 topics across 15 phases (added math for ML, deeper statistics/experimentation, NLP, computer vision, data engineering, cloud ML platforms, and responsible AI/ethics). Both now also carry a `TOPIC_RESOURCES` map (matching `java-backend.js`'s pattern) with curated links for signature topics.
- **Four new starter templates, diversifying beyond software engineering** (issue #51 follow-up): `genai-agentic-ai.js` (GenAI/Agentic AI Engineer — LLM fundamentals, prompt engineering, RAG, agents, frameworks, fine-tuning, evaluation, safety; 81 topics across 11 phases), `math-grade12.js` (12th Grade Mathematics — relations/functions, algebra, calculus, vectors/3D geometry, linear programming, probability, exam prep; 85 topics across 7 phases), `piano.js` (Learning Piano — reading music, technique, theory, repertoire, sight-reading, ear training, performance; 65 topics across 8 phases), and `marketing.js` (fundamentals, branding, content/SEO, paid ads, social/email, analytics, growth, strategy, career; 75 topics across 11 phases). The onboarding picker now offers 8 templates in total.
- **Per-user "hide this template" preference** (issue #51 follow-up): every template except "blank" now has a small hide (×) button on its onboarding card. Hiding asks for confirmation, then removes the card from *that user's* picker only — it's stored in `users/{uid}/meta/hiddenTemplateIds` (plus a local fallback), never affects the template's content, and never affects any other user. A "Show hidden templates" toggle appears whenever the user has hidden at least one, with a "Restore" action per hidden card. New `roadmapStore.hideTemplate(id)` / `unhideTemplate(id)`; `dbApi`/database rules extended for the new `meta.hiddenTemplateIds` field (array of strings).
- **"Build your own roadmap" guide** (issue #51 follow-up): the "Start blank" card has an "ℹ How do I build my own?" button (in place of the hide button, since blank can never be hidden — it's the gateway to a self-built roadmap) that opens a modal explaining how to add topics manually via the dashboard's "Add a custom topic…" row, plus a practical tip for using an external AI chat assistant to draft topics and paste them in one at a time. Fully automated AI import is noted as a planned future feature, not implemented yet — the modal doesn't overpromise.
- **Custom `confirmDialog()` component, replacing every native `window.confirm()`** (issue #51 follow-up): the browser's built-in confirm dialog looked out of place next to the rest of the app's themed UI. New `src/ui/components/confirmDialog.js` renders a `Promise<boolean>`-returning modal using the same `.modal-overlay`/`.modal-card` styling as the existing delete-account and build-your-own-roadmap modals, with Escape-to-cancel and click-outside-to-cancel. Replaces `confirm()` in all four places it was used: switching templates, hiding a template, signing out with unsaved guest changes, and deleting a topic. Destructive/irreversible actions get a red "danger" confirm button.
- **Clicking the "Ascent" logo now goes home** (issue #51 follow-up): the brand mark in the dashboard header and onboarding picker was a non-interactive `<div>` — clicking it did nothing, unlike the sign-in/sign-up pages where it already linked back to `#/signin`. Both now link to `#/onboarding` (the "all roadmaps" picker), matching the existing pattern.
- **"Which roadmap am I on?" is no longer ambiguous** (issue #51 follow-up): the dashboard hero now shows a badge with the active template's icon and name above the main heading. The "Switch your starter roadmap" picker marks the currently-active template's card with a highlighted border and a "Current" badge. Re-picking that same card no longer asks for confirmation or re-seeds the roadmap (which previously would have silently discarded progress on the template the user was already on) — it just returns to the dashboard unchanged.

### Fixed
- **The previous template's content could silently reappear after switching roadmaps** (issue #51 follow-up, data-integrity bug): reported from real usage — switching to a different starter template would sometimes show the *previous* template's items (including checked-off progress) instead of the freshly-picked one, reproducing on nearly every switch. Root cause: `initFromTemplate()` resets local state synchronously but can't cancel a debounced save from the previous template that's already in flight to Firebase; that save's echo arrived after the switch, was no longer recognized as an echo (`lastFlushedStr` had been reset), and got applied as if it were genuinely new data — overwriting the new template's items with the old template's, both in memory and in `localStorage`. Fixed by tagging every saved roadmap payload with the `templateId` it was written for and having the realtime listener reject any incoming update whose `templateId` doesn't match the currently active template. `firebase/database.rules.json` updated to validate the new field. 4 new regression tests in `tests/integration/roadmapStore.test.js` deterministically replay the exact race (verified to fail without the fix, pass with it) rather than relying on flaky real-time timing.
- **Stale `setUser()` calls could clobber a newer sign-in's state** (issue #51 follow-up): Firebase's `onAuthStateChanged` can fire in quick succession — most notably deleting an account and immediately signing up again with the same email. Because `setUser()` does an async Firebase round-trip before resolving onboarding status, a slower/older call that was still in flight could finish *after* a newer call and overwrite its correct state with stale data, which could manifest as a freshly-recreated account skipping the onboarding picker. `roadmapStore.js` now tracks a `stateCallId` generation counter (shared with `initFromTemplate`) and any call whose id has been superseded by a newer one aborts without mutating state after its `await`s resolve.
- **Onboarding template cards rendered at inconsistent heights within the same row** (issue #51 follow-up): `.template-card` had no `height: 100%` and used `display: grid` with content-sized rows, so a card's box only ever sized to its own text — cards with a longer description or an extra footer button (e.g. the "Start blank" card's "How do I build my own?" link) visibly stuck out below their row-mates instead of matching them. `.template-card` is now a `height: 100%` flex column so it fills the grid's stretched row height, and `.template-card-count` gets `margin-top: auto` so the topic-count footer stays flush to the bottom of every card in a row regardless of description length. `.template-card-hidden` (the "Show hidden templates" restore cards) gets the same treatment. Also fixed a regression introduced while making this change: switching from `display: grid` (which used `justify-self: start`) to `display: flex` briefly caused the "How do I build my own?" button's text to center itself (the browser default for `<button>`), fixed with an explicit `align-self: flex-start; text-align: left`. Verified visually at 390px/820px/1440px viewport widths. See the new "Card/grid layout" convention in `CLAUDE.md`/`AGENTS.md` for the pattern going forward.

"Delete account" button added to the dashboard header for authenticated (non-guest) users, opens a confirmation modal with password re-entry. - **Forgot password / password reset** (issue #13): "Forgot password?" link added below the Password field on the sign-in page. Success state shows a confirmation message and never leaks whether an account exists for the submitted email (non-existent accounts show the same success UI). Uses Firebase's default hosted action URL (Option A) — no custom reset-confirm form. `authApi.sendResetEmail(email)` added to `src/services/firebase.js`. `authShell` now exposes `titleEl` and `subtitleEl` so pages can update the card header when switching inline views.
