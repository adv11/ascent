## [Unreleased]

### Added
- **Cross-device / responsive consistency overhaul** (issue #36): replaced the old ad-hoc `920px`/`640px` breakpoint pair with a documented six-tier scale (`≤375` small phone / `≤480` phone / `≤768` tablet portrait / `≤1024` tablet landscape / untuned laptop–desktop base / `≥1600` ultra-wide) — see `docs/adr/ADR-006-responsive-breakpoints-touch-hover.md`. Fixed `.check-actions` (the per-row Edit/resource controls) being hidden on any touch-capable device at a desktop-sized viewport (touchscreen laptop, Surface, iPad landscape) by switching its reveal condition from a `max-width` proxy to `@media (hover: none), (pointer: coarse)` — the correct signal for "does this input device have a reliable hover." Touch targets (`.btn-icon`, `.btn-sm`, `.filter-chip`, checklist rows) are raised to a ~44×44px minimum (WCAG 2.5.5) under the same `(pointer: coarse)` condition. Replaced every `100vh` full-height container (`.app-shell`, `.auth-page`, `.onboarding-page`, `.dashboard`) with a `100vh`-then-`100dvh` pair so fixed-height layout no longer gets cut off or jumps when a mobile browser's address bar collapses/expands. Added a `≤1024px` override bumping every focusable field (`.field-input`, `.field-input.compact`, `.search-input`, `.import-paste-area`) to 16px, preventing iOS Safari's auto-zoom on focus. Added `viewport-fit=cover` to `index.html`'s viewport meta and `env(safe-area-inset-*)` padding to the dashboard header, auth/onboarding pages, the item panel, and toast/save-badge chrome, so an installed (`"display": "standalone"`) PWA session isn't obscured by a notch/Dynamic Island/gesture bar. Added a `-webkit-tap-highlight-color: transparent` reset. Documented a cross-device verification matrix (11 widths, touch/hover checks, real iOS Safari + Android Chrome passes) in `CLAUDE.md`/`AGENTS.md`'s "Verifying changes" section.
- **Firebase Database Rules hardening + anonymous data cleanup** (issue #24): `firebase/database.rules.json` now validates `version`/`updatedAt` as numbers (previously only presence-checked via `hasChildren`) on both the legacy `roadmap` node and `roadmaps/$templateId`, adds a `$other: { ".validate": "false" }` catch-all under `users/$uid` so a write to any path other than `roadmap`/`roadmaps`/`meta` is rejected, and adds a `reports/$uid` stub (write auth-scoped, read denied) ahead of issue #9 actually using it. Since Realtime Database rules can't count a map's children, the 1,000-item-per-roadmap soft cap is enforced client-side instead — `roadmapStore.js`'s `addItem()` returns `false` once a roadmap already holds 1,000 non-deleted items, and the dashboard's "Add a custom topic…" row shows an error toast instead of silently doing nothing. Unlinked guest sessions no longer leave orphaned data behind: `authApi.signOut()` now deletes an anonymous user's database node and their Firebase Auth record (instead of a plain sign-out) when they exit without linking to a real account — see `docs/adr/ADR-005-anonymous-user-lifecycle.md` for the options considered and why a Cloud Function TTL sweep wasn't chosen (Blaze-plan-only; this project targets the free Spark plan).
- **Personal notes per topic** (issue #15): every roadmap item gains a `notes` field (plain text, capped at 5,000 characters) alongside its existing `resources`. The item panel (`src/ui/components/itemPanel.js`) shows a new Notes section between Priority and Resources — an auto-growing textarea (`field-sizing: content`) that autosaves 800ms after you stop typing (no need to hit "Save changes"), with a fading "Autosaved ✓" indicator and an amber "Failed to save — check connection" message if the save callback throws. A character counter appears only once you're past 80% of the limit; the textarea's native `maxlength` caps input at the DOM level, not just on save. A pending autosave is flushed immediately if you close the panel before the debounce fires, so an edit made in that narrow window is never silently dropped. A roadmap row with a non-empty note shows a small 📝 indicator (`data-action="notes"`, native `title` tooltip) next to its resource badge; clicking it opens the item panel with focus already in the notes textarea. `roadmapStore.js`'s `updateItem` already treated any non-`done` patch as structural, so a notes edit was non-cosmetic (bumping `structuralVersion` so the indicator re-renders) with no code change needed there — see the new comment guarding against ever adding `'notes'` to that cosmetic-check. Existing items with no `notes` field read as `''`, no migration needed.
- **Retired the "Start blank" starter template** (issue #4 follow-up): once manual roadmap creation and AI-assisted import (both above) existed, "blank" — four fixed, uneditable phases, exactly one per account, never hideable — became a strict subset of "Create your own roadmap" (fully editable, unlimited instances, deletable). Removed from `TEMPLATES` (down to 7 built-in templates); every remaining built-in template card now has a hide button with no exceptions. Anyone who already started "blank" is migrated forward automatically on their next sign-in, into a real custom roadmap ("My roadmap") preserving their exact items and phases — the old Firebase path is never deleted, just no longer read from. The "build your own" guide moved from blank's info button onto "Create your own roadmap"'s new corner ℹ button, and was rewritten to cover both the manual workflow and the real AI-import flow (an "Open Import roadmap" button hands off directly) instead of the old copy-topics-in-one-at-a-time tip.
- **AI-assisted roadmap import** (issue #4): a second "Import roadmap" entry point next to "Create your own roadmap" — a two-tab modal (`src/ui/components/importRoadmapModal.js`). "Generate with AI" shows a versioned, copyable prompt (`src/data/importPrompt.js` — `buildImportPrompt(topic)`, `IMPORT_PROMPT_VERSION`) with a live-updating editable topic line and a "Copy prompt" button (Clipboard API with an `execCommand('copy')` fallback). "Paste & Import" validates pasted AI output 300ms after each keystroke via a new pure validator, `src/core/roadmap/importValidator.js` (`validateImportText`/`validateImportPayload`/`parseImportJson`) — checks schema version, required fields, priority enums, and a 500-item cap, surfacing per-field error messages (e.g. `phases[2].sections[0].items[1] is invalid`). Once valid, a second pure module, `src/core/roadmap/schemaAdapter.js` (`adaptImportToRoadmap`), converts the validated JSON into the internal `{ phases, items }` shape and the "Import roadmap" button enables. Importing calls the exact same `store.createCustomRoadmap({ title, phases, items })` the manual "Create your own roadmap" flow already used (issue #4 CRUD) — just with the phases/items pre-populated instead of empty — via a new one-shot `pendingCustomSeeds` mechanism in `roadmapStore.js`, so an imported roadmap gets identical Firebase persistence, phase/section rename/delete controls, and delete-roadmap cleanup to a manually-built one. Validator and adapter are deliberately separate pure functions (no DOM/store/Firebase) so a future schema version only ever means adding a new adapter, never touching validation rules.
- **Manual "Create your own roadmap" — full phase/section/topic CRUD** (issue #4): users are no longer limited to the 8 built-in starter templates — a new "Create your own roadmap" card (always first in the `/onboarding` grid) opens `src/ui/components/newRoadmapModal.js` to collect a title and optional description, then activates a brand-new, completely empty roadmap through the exact same `switchRoadmap()` path a built-in template uses. `roadmapStore.js` gains a `createCustomRoadmap`/`deleteCustomRoadmap` pair and a `customRoadmaps` list (`{ id, title, description, createdAt }`, synced to `users/{uid}/meta.customRoadmaps`) tracking every roadmap a user has built manually — each gets a generated `croadmap-<timestamp>-<random>` id (`isCustomRoadmapId()`) instead of a built-in template id, and flows through the same per-template Firebase path, `startedTemplateIds`, and cache-first resolution issue #58 built for templates. Unlike a built-in template's fixed phase/section skeleton (defined in code, never persisted), a custom roadmap's phases/sections are fully mutable and are the only record of what the user built — new `addPhase`/`renamePhase`/`removePhase`/`addSection`/`renameSection`/`removeSection` store methods (silent no-ops on a built-in template) persist and resolve them exactly like `items` already were, including the same Firebase-echo/multi-device-sync guard (now comparing `{ items, phases }` together instead of `items` alone). `dashboard.js` shows "+ Add phase"/"+ Add section" controls and inline rename/delete for phases and sections only when viewing a custom roadmap; renaming re-files every affected item to the new title, removing soft-deletes every item under the removed phase/section. The onboarding picker renders a card per custom roadmap (with the same Current/In-progress badges as built-in templates) and a delete (×) button that permanently removes it (Firebase node, local blob, and meta entry) after confirmation — switching away to the default built-in template first if it was active. `firebase/database.rules.json` extended with a `meta.customRoadmaps` array validator. Superseded the "default pinned/read-only roadmap" half of the original issue #4 proposal: with the starter-template system (issue #51) already giving every user a roadmap on first load, a separate read-only "default" concept would have been redundant — every built-in template already behaves as a shared, always-available starting point. The AI-assisted import flow described in the original issue ships as its own entry above.

### Fixed
- **A local edit could be silently reverted if the page reloaded before its debounced Firebase write landed** (issue #67): `resolveRoadmapItems` (`roadmapStore.js`, used on every sign-in/page load) used to prefer a successful Firebase read over the local cache unconditionally, whenever Firebase had any stored data — even if the local blob was marked `dirty` (an edit queued or in-flight when the page was torn down). A reload right after an edit could beat the 500ms save debounce, leaving Firebase holding stale pre-edit data that then silently overwrote the fresher local copy on the next load. Found via a deterministically-failing E2E test (`tests/e2e/itemNotes.test.js`'s "notes survive a page reload") reproduced against a local Firebase emulator on a clean `main` checkout. Fixed by having `resolveRoadmapItems` check the local blob's `dirty` flag before ever attempting a remote read — mirroring the same guard `attachRoadmapListener` already had for the realtime-listener path (issue #58).

### Removed
- **Google Sign-In and the Google Drive storage backend** (issues #5, #71): after two days spent chasing real-world OAuth issues (popup timing, CSP, GIS silent refresh) without a version reliable enough to ship, the decision was made this isn't worth the ongoing implementation/maintenance cost right now. `GoogleDriveAdapter` and its "Sign in with Google" UI/token wiring are fully removed (code, tests, docs); Firebase remains the only storage backend and sign-in method. Issues #5 and #71 are closed. The generic `StorageAdapter` interface (below) is kept, since it's now load-bearing for other shipped features and behaves identically to hard-wiring Firebase directly.

### Changed
- **Storage adapter abstraction — internal refactor, no user-facing change** (issue #5, part 1): `roadmapStore.js` no longer imports `firebase.js` directly for roadmap/meta reads and writes — it now goes through a `StorageAdapter` interface (`src/services/storage/`). `FirebaseAdapter` carries over the exact previous Realtime Database logic (formerly `dbApi`) with no behavior change; `adapterFactory.js`'s `getStorageAdapter()` always returns it, so every signed-in account (including guest/anonymous sessions) behaves identically to before. A standalone `LocalStorageAdapter` implementing the same contract also ships in this PR but isn't wired in yet — it's scaffolding for a future guest-only/offline-cache mode. The interface's shape was adapted from a naive single-roadmap sketch to match the app's real multi-user/multi-template/meta data model (see `docs/architecture.md` §5.12).

### Added
- **Multi-roadmap support — concurrent progress per template, non-destructive switching** (issue #58): previously the app had exactly one roadmap slot per account (`users/{uid}/roadmap`), and switching starter templates always replaced it wholesale — a user tracking Frontend for a job search and Piano as a hobby at the same time couldn't keep both moving, since picking one destroyed the other. Each started template now gets its own Firebase node, `users/{uid}/roadmaps/{templateId}/`, tracked in `users/{uid}/meta.startedTemplateIds` with `meta.activeTemplateId` naming the one currently displayed. `roadmapStore.js`'s `switchRoadmap(templateId)` replaces `initFromTemplate()` — an already-started template loads its own saved progress (cache-first: in-memory this session, then Firebase, then local storage, then seed), a not-yet-started one seeds fresh, and neither ever touches another template's data. The onboarding picker (`src/ui/pages/onboarding.js`) drops its "this replaces your current roadmap" confirmation entirely — switching is now always instant and non-destructive — and gains a third card badge state, "In progress", for a template that's been started but isn't currently active; a started template that's also hidden stays visible (badged) instead of disappearing, since hiding only ever filters the "start something new" grid. Locally, `KEYS.ROADMAPS` (`ascent-roadmaps-v1`) replaces the single `KEYS.ROADMAP` blob with a per-template keyed shape; existing local data is migrated forward automatically, leaving the old key in place. Existing (pre-#58) Firebase accounts are migrated transparently on their next sign-in — the old singular `users/{uid}/roadmap` path is copied into the new per-template path and left in place as a safety net, never deleted. `firebase/database.rules.json` extended with the `roadmaps/$templateId` node and `meta.activeTemplateId`/`meta.startedTemplateIds` validators, alongside (not replacing) the legacy validators. Three correctness fixes specific to per-template Firebase paths: a debounced edit pending on the template you're switching *away from* is now flushed to its own path before the switch, instead of risking being silently dropped once the timer later fires against the newly active template; a closure-based "stale listener guard" (each template's realtime listener is scoped to the exact template it was attached for) replaces the old #51 cross-template echo payload tag, which is no longer needed now that each template has its own path; and (found via real E2E testing against live Firebase, not by the mocked test suite) the realtime listener now never applies an incoming snapshot while a local edit is still unflushed (`dirty === true`) — a queued-or-in-flight edit is always newer than anything the listener could be echoing, and Firebase's own data normalization means a delayed echo of an older write doesn't always byte-for-byte match what was sent, so string-matching alone could occasionally misclassify a stale echo as "genuinely newer" and silently revert an edit made in the narrow window right after starting a fresh (not-yet-started) template.

### Fixed
- **A brand-new guest/sign-up could get stuck bouncing between `/onboarding` and `/app` without ever starting a roadmap** (issue #58 follow-up, regression from the multi-roadmap work): "Continue as guest" (and every sign-in path) navigates to `/onboarding` immediately after the Firebase auth call resolves, without waiting for `roadmapStore.setUser()` — which is still resolving its own async Firebase read at that point. `renderOnboarding` could therefore mount using the store's pre-`setUser()` placeholder default (`activeTemplateId = 'java-backend'`), not the fresh user's real state. Picking the Java Backend Engineer card then matched that placeholder and hit `pickTemplate`'s "you're already on this roadmap" shortcut, which just navigated to `/app` without ever calling `switchRoadmap()` — leaving `onboardingDone` still `false`, so the dashboard's own guard immediately bounced back to `/onboarding`. Fixed by only treating a same-id pick as a no-op once onboarding was already done at render time (`isSwitchingTemplate`), so a first-time pick always calls `switchRoadmap()` regardless of the placeholder default. Also fixed a stale assertion in `tests/e2e/onboarding.test.js`'s multi-template regression test that checked a template's "In progress" badge before it had actually stopped being the active one.
- **The previous template's content could silently reappear after switching roadmaps** (issue #51 follow-up, data-integrity bug): reported from real usage — switching to a different starter template would sometimes show the *previous* template's items (including checked-off progress) instead of the freshly-picked one, reproducing on nearly every switch. Root cause: `initFromTemplate()` resets local state synchronously but can't cancel a debounced save from the previous template that's already in flight to Firebase; that save's echo arrived after the switch, was no longer recognized as an echo (`lastFlushedStr` had been reset), and got applied as if it were genuinely new data — overwriting the new template's items with the old template's, both in memory and in `localStorage`. Fixed by tagging every saved roadmap payload with the `templateId` it was written for and having the realtime listener reject any incoming update whose `templateId` doesn't match the currently active template. `firebase/database.rules.json` updated to validate the new field. 4 new regression tests in `tests/integration/roadmapStore.test.js` deterministically replay the exact race (verified to fail without the fix, pass with it) rather than relying on flaky real-time timing.
- **Stale `setUser()` calls could clobber a newer sign-in's state** (issue #51 follow-up): Firebase's `onAuthStateChanged` can fire in quick succession — most notably deleting an account and immediately signing up again with the same email. Because `setUser()` does an async Firebase round-trip before resolving onboarding status, a slower/older call that was still in flight could finish *after* a newer call and overwrite its correct state with stale data, which could manifest as a freshly-recreated account skipping the onboarding picker. `roadmapStore.js` now tracks a `stateCallId` generation counter (shared with `initFromTemplate`) and any call whose id has been superseded by a newer one aborts without mutating state after its `await`s resolve.
- **Onboarding template cards rendered at inconsistent heights within the same row** (issue #51 follow-up): `.template-card` had no `height: 100%` and used `display: grid` with content-sized rows, so a card's box only ever sized to its own text — cards with a longer description or an extra footer button (e.g. the "Start blank" card's "How do I build my own?" link) visibly stuck out below their row-mates instead of matching them. `.template-card` is now a `height: 100%` flex column so it fills the grid's stretched row height, and `.template-card-count` gets `margin-top: auto` so the topic-count footer stays flush to the bottom of every card in a row regardless of description length. `.template-card-hidden` (the "Show hidden templates" restore cards) gets the same treatment. Also fixed a regression introduced while making this change: switching from `display: grid` (which used `justify-self: start`) to `display: flex` briefly caused the "How do I build my own?" button's text to center itself (the browser default for `<button>`), fixed with an explicit `align-self: flex-start; text-align: left`. Verified visually at 390px/820px/1440px viewport widths. See the new "Card/grid layout" convention in `CLAUDE.md`/`AGENTS.md` for the pattern going forward.
