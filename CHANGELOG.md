# Changelog

All notable changes to this project will be documented in this file.

## [Unreleased]

### Added
- **Printed/exported-to-PDF roadmaps now show a faint, blurred Ascent logo+wordmark watermark behind the checklist content on every page — real feedback (screenshot).** `printRoadmap.js` appends a `position: fixed` `.print-watermark` layer alongside the existing print snapshot (repeats correctly per page on its own, same as `position: fixed` already does for one-off elements — see the header/footer entry below for why that mechanism isn't right for content that needs *reserved* space, which the watermark deliberately doesn't). Low opacity (0.07) plus a slight blur keeps the roadmap's own checklist text fully legible on top of it, per the explicit "both should be visible" requirement.

### Changed
- **The "Share your progress" social card (`shareCard.js`) still used the app's old, retired pre-Alpenglow teal gradient — real feedback (screenshot).** `--brand-500/600/700` were carved out as a deliberate exception when the rest of the app moved to the gold→rose "Alpenglow" identity (issue #155 v2/#206), on the reasoning that this canvas-rendered card was a distinct product-identity surface — but that left it as the one remaining place in the app still showing the retired look. `drawBackground()` now uses `--color-brand-gold-ink`/`--color-brand-rose-ink` (the same darkened variants already used elsewhere for text-on-light-fill contrast, e.g. `.template-card-ai-badge`) rather than the base `--color-brand-gold`/`--color-brand-rose` tokens — the base tokens are light/mid-toned (tuned to host *dark* ink text, like `.btn-primary`), and would fail contrast for this card's white text the same way the old dark teal-700→teal-500 stops never did. The now-fully-unused `--brand-50/100/500/600/700` teal scale was removed from `app.css` rather than left as dead tokens.

### Fixed
- **Printed roadmap footers hardcoded `ascent-app.com` — real feedback: this should update automatically if the production domain ever changes, and it also printed the wrong URL (`localhost:4173`) during local dev.** `printRoadmap.js`'s footer text now reads `window.location.host` at print time via a small `printSiteUrl()` helper, instead of a literal `PRINT_SITE_URL` constant — it always reflects wherever the app is actually being served from (dev, a PR preview, or production) with nothing to keep in sync by hand.
- **The sign-in/sign-up marketing panel's dark background never actually painted — its own white headline/subhead/feature-list text rendered as barely-visible pale ghosting on the light page background instead of white-on-near-black, on every auth page — real bug, reported with a screenshot.** `.auth-page-bg` (`app.css`) is a light `--color-bg`-filled layer meant to sit behind just the right-hand sign-in card column (`.auth-page-right`) — but it was `position: fixed; inset: 0`, which sizes itself against the full viewport regardless of DOM nesting, not its parent. On the two-column auth layout this painted the light fill across the *entire* page, including over the always-dark `.auth-marketing` panel on the left. `pointer-events: none` (needed so the layer never blocks clicks) also hid this from click/hit-testing tools, which is why it went unnoticed as a functional bug — nothing was ever unclickable, only visually broken. Changed to `position: absolute`, which correctly scopes `inset: 0` to `.auth-page-right` (already `position: relative`) instead of the viewport.
- **A `custom-select`/`dropdown-menu` left open while the page scrolled rendered as a giant stuck overlay covering unrelated page content — real bug, reported with screenshots from the onboarding page's Daily Todos duration select.** `select.js`'s listbox and `dropdown.js`'s menu are both `position: fixed` elements portaled to `document.body`, positioned once (from the trigger's `getBoundingClientRect()`) at the moment they open. `position: fixed` doesn't track page scroll, so scrolling the page while either was open left it visually anchored at its original screen coordinates while the trigger — and everything else — scrolled out from under it, with no repositioning or auto-close to correct it. A first fix (close on any `scroll`/`resize` event) caused a real regression caught by CI's E2E suite: this app's global `scroll-behavior: smooth` turns an unrelated `.focus()` elsewhere on the page (e.g. a modal opening) into a multi-hundred-millisecond stream of genuine scroll events, which an any-scroll-closes listener misread as the user scrolling the dropdown away — closing it before a test (or a real user) ever interacted with it. The real fix: both components now track the trigger's on-screen position at open time and only close once it has moved more than a few pixels, absorbing unrelated scroll jitter while still closing promptly on a real page scroll; `{ preventScroll: true }` was also added to every focus call inside both components so they never contribute to that jitter themselves. `select.js`'s listener additionally ignores scroll events whose target is the listbox itself, since `.custom-select-listbox` is `overflow-y: auto` for long option lists.
- **Printed PDFs showed the browser's own injected chrome — the page URL (`localhost:4173/#/app` during local dev) and date/time in the corners of every page — real feedback (screenshot).** This is Chrome/Firefox's own default print header/footer, rendered into the blank margin band the browser reserves around the page by default; it isn't something `app.css` previously addressed at all, and in production it would have shown `ascent-app.com` rather than an actual bug — but it's confusing and unpolished either way, and a user's own Chrome "Headers and footers" print-dialog setting is outside this app's control. Adding `@page { margin: 0; }` removes the blank margin band that chrome renders into, in both engines, regardless of that per-user toggle. Since the browser's default margin no longer exists, `.print-roadmap`'s `<thead>`/`<tfoot>`/`<tbody>` cells now carry their own explicit `padding: 0 40px` (sides) so content doesn't run edge-to-edge on the page.
- **Printed/exported-to-PDF roadmaps overlapped their own repeating brand header on every page after the first — real bug, reported with a live 47-page multi-phase PDF export.** `printRoadmap.js`'s `.print-page-header`/`.print-page-footer` used `position: fixed`, which does repeat on every physical page in Chrome, but the clearance for them relied on `.print-roadmap`'s own `padding: 64px 0 56px` — which only reserves space once, at the top/bottom of the whole flowed document, not per physical page. Every page after the first rendered its heading text flush against the page edge, directly underneath the fixed header. An `@page { margin-top/-bottom }` rule was tried next (margins genuinely do re-apply per page) but a real multi-page `page.pdf()` render with direct text-position inspection showed the same overlap persisted — Chromium positions `position: fixed` print content relative to the page's content box (already inside the margin), not the physical page edge, so the fixed header still landed at the same coordinate the flowing content starts at. The real fix: `printRoadmap.js` now builds a genuine `<table>`/`<thead>`/`<tfoot>`/`<tbody>` tree instead of `<div>`s with a `position: fixed` class — the browser's native table-header/footer print-repeat behavior both repeats the branded header/footer *and* correctly reserves space for it on every page, confirmed via a real multi-page PDF render. See `.claude/rules/ui-styling.md`'s "Branded print/PDF export" entry for the full writeup of all three approaches tried.
- **CI's "Lighthouse CI (perf budget)" job was intermittently failing with `NO_FCP` ("The page did not paint any content"), unrelated to any real regression.** Confirmed the landing page paints correctly and fast (~20ms) locally with a clean `localStorage`, so this wasn't a genuine app bug — the CI runner's headless Chrome occasionally missed the previous `maxWaitForFcp: 60000` budget under load, and a single `numberOfRuns: 1` collection meant one slow run failed the whole job outright. `lighthouserc.json` now runs 3 collections (`numberOfRuns: 3`, giving LHCI a median rather than a single sample) and raised `maxWaitForFcp`/`maxWaitForLoad` to 120000/150000 for headroom on a busy runner.
- **The checked-checkbox fill still read as green after a first warm-olive retune — real feedback (screenshot), a second round.** Retuning the hue wasn't enough since it's still a fundamentally different color family from gold/rose. `.check-box.done` now uses a flat `var(--color-brand-gold)` fill instead of any green, matching every other "selected/active/complete" indicator in the app (the active-roadmap badge, the sidebar's active-nav bar, the progress ring). The checkmark icon switched from fixed white to `--color-ink-on-gold` (dark ink) to avoid the documented white-on-gold dark-theme contrast failure. `--color-success`/`-bg` tokens are unchanged and still used elsewhere (e.g. `.toast-success`'s accent bar) — only the checkbox moved off them.
- **A CSS comment silently deleted the entire `.toast` rule from the stylesheet — the actual root cause of the "toast barely visible" report, not just a contrast issue (issue #206 §5 follow-up).** A comment added earlier in this same pass read `"only --duration-*/--ease-* tokens"` — the literal substring `--duration-*/--ease-*` contains `*/`, which is the CSS comment-close token. The parser closed the comment mid-sentence right there, and everything after it up to the next real `}` (the rest of that sentence, plus the *entire* `.toast { ... }` rule that followed) was consumed as one broken, silently-dropped construct — confirmed by inspecting the parsed `CSSStyleSheet` directly (`.toast-stack` and `.toast.show` both present; the base `.toast` rule completely absent, despite the raw file text being intact). This meant `.toast` had **zero** styling applied at all — no background, border, shadow, padding, or border-radius — which is what actually produced the "plain unstyled text" toast in the screenshot, not merely a contrast problem with an otherwise-styled card. Rewrote the comment to avoid the literal `*/` sequence. Also added a real 1px `--color-border` outline to `.toast` (on top of the fix above) for better definition against `--color-bg`, since the two are visually close in light theme.
- **The checked-checkbox fill and other `--color-success`-derived colors read as a jarring, out-of-family cool green against the warm gold/rose/cream Alpenglow palette — real feedback (screenshot).** Retuned `--color-success`/`--color-success-bg` warmer/more olive in both themes (same lightness, yellow-shifted hue) so the app's single highest-frequency colored surface (`.check-box.done`) reads as part of the same warm family while staying a real, distinct green from `--color-brand-gold`.
- **The feedback/feature-request modals' Severity and "How often would you use it?" radio groups rendered as unstyled OS-native radio buttons inside a bare browser-default `<fieldset>` box (a visible inset-groove border no other field in the app has) — real feedback (screenshots).** `.feedback-radio-fieldset` resets the fieldset/legend UA defaults; the radio inputs get a custom `appearance: none` circle matching `.check-box`'s visual language (border → filled gold dot on check). The "Include system info" checkbox in the same modals gets `accent-color: var(--color-brand-gold)`, the same convention `.remember-checkbox`/`.daily-todo-checkbox` already use, replacing an unstyled native blue check.
- **`.template-card-ai-badge`'s "AI-powered" text failed WCAG AA contrast (3.31:1, needs 4.5:1) — a real, axe-confirmed bug, not a sampler artifact (issue #206 §5 follow-up).** Plain `var(--color-brand-rose)` text on `var(--color-danger-bg)` measured 3.31:1 in light theme. Added `--color-brand-rose-ink` (light: `color-mix()`-darkened to 7.73:1; dark: rose already clears 4.5:1 as a passthrough) following the same pattern `--color-brand-gold-ink` already established, and switched the badge to it.
- **CI's automated accessibility suite picked up a batch of axe sampler-misattribution false positives after the ambient-gradient removal above changed the pixel neighborhood around several unrelated elements.** Each one was live-verified (via `getComputedStyle()` or a grep for the reported hex against `app.css` — no match in every case) to not be a real contrast bug before being added to `tests/e2e/accessibility.test.js`'s `CONTRAST_FALSE_POSITIVE_SELECTORS` list — see that file's own comment for the full per-selector verification. Also fixed a genuine new test flake: the overflow-menu interaction (§4.1) turned a single click into two sequential clicks, and `onboarding.js`'s live store-snapshot subscription can re-render the card grid in the narrow window between them, closing the just-opened menu before the second click lands — every e2e test driving the overflow menu now retries the open+click pair via `expect(...).toPass()`.
- **The feedback widget's "Capture current screen" button always failed with "Could not capture the screen" — a pre-existing bug, unrelated to any recent design change.** `screenshotCapture.js`'s `captureScreenshot()` uses `html2canvas`, which by default clones the DOM and copies each cloned node's computed styles onto its own inline `style` attribute so the detached clone renders identically before being rasterized — this app's Content Security Policy has no `unsafe-inline` for `style-src` (a deliberate hardening choice, see `.claude/rules/auth-security.md`), so the browser silently dropped every one of those inline styles, breaking the capture on every attempt. Fixed by passing `foreignObjectRendering: true` to `html2canvas()`, which renders the live DOM directly via an SVG `<foreignObject>` using the browser's own rendering engine instead of the clone-and-inline-restyle approach — needs no inline styles at all, so the CSP no longer interferes. No CSP change was needed.

### Security
- **`users/{uid}/reports/{reportId}`'s validation rule didn't mirror the top-level `reports/{reportId}` rule's type/length checks (issue #192).** `submitReport()` (`src/services/feedbackStore.js`) writes the same report to two paths in one multi-path update: the developer-only-readable `reports/{reportId}` (which already enforced a `type` regex of `bug|feature|feedback` and a 120-char `title` cap) and the per-user history mirror `users/{uid}/reports/{reportId}` (which only checked `hasChildren(['type','title','submittedAt'])`, with no type enum and no length cap). A client bypassing `feedbackStore.js` entirely could write arbitrary-length garbage or an invalid `type` into their own report history — self-scoped, not a cross-user leak, but a real rule inconsistency for what's logically the same write. `firebase/database.rules.json`'s per-user copy now enforces the identical `type`/`title` shape as the top-level rule. Verified against the Firebase emulator in `tests/e2e/dataCapRules.test.js`.
- **`sharedRoadmaps/{shareId}` — the one publicly, unauthenticated-readable path in the app — had a completely unvalidated `items` field (issue #188).** `firebase/database.rules.json`'s `.validate` rule for a shared roadmap snapshot checked `schemaVersion`/`ownerUid`/`templateId`/`title`/`publishedAt` but placed no constraint at all on `items`: no type check, no per-field size cap, and it wasn't even in the required-children list — any authenticated user (including a guest account) could publish a share node with an arbitrarily large or malformed `items` payload onto the one path anyone on the internet can read, consuming database quota on a public-facing surface with no gate. `items` is now required, and each item must match the exact narrow shape `shareSchema.js`'s `toShareItem()` emits (`title`/`phase`/`section` ≤200 chars, `priority` ≤50, `done` a boolean, an optional `resources` list capped like every other resource list in the file) with any field outside that set rejected outright. See `.claude/rules/auth-security.md` for the full rule writeup and the accepted item-*count* limitation (RTDB rules still can't cap `numChildren()`, same gap issue #122 already accepted elsewhere). Verified against the Firebase emulator in `tests/e2e/dataCapRules.test.js`.
- **`roadmaps/$templateId`'s `items` and `phases` had no per-field server-side schema validation, and neither `roadmaps/$templateId` nor `meta` rejected undeclared keys (issue #187).** Issue #122 added length/shape caps for `title`/`notes`/resource `label`/`url` under `roadmaps/$templateId/items/$itemId`, but left the rest of an item's shape (`priority`, `phase`, `section`, `done`, `custom`, `deleted`, `createdAt`) and the resource-array size completely open, and `phases` (a custom roadmap's own phase/section structure) had zero validation at all — any authenticated user could write an arbitrarily large or malformed `phases` array, or set `priority` to any string, directly against their own uid-scoped path. `firebase/database.rules.json` now validates every item field (title non-empty and ≤200, `priority` restricted to the `P0`-`P3` enum `roadmapStore.js`/`importValidator.js` already use, `done`/`custom`/`deleted` booleans, `createdAt` a number) and caps `resources` to at most 100 entries via the same index-whitelist idiom `favoriteRoadmapIds`/`meta.customRoadmaps` already use; `phases` is capped at 1000 entries, each with an id, a required ≤200-char `title`, an optional `P0`-`P3` `priority`, and up to 1000 `sections`, each with its own id and required ≤200-char `title`. `roadmaps/$templateId` and `meta` both gained a `$other: { ".validate": "false" }` catch-all, matching the existing pattern at `users/$uid`, so an undeclared top-level key can no longer be written to either path. Verified against a real local Firebase emulator (`tests/e2e/dataCapRules.test.js`), including that every existing custom-roadmap/AI-import/manual-CRUD e2e flow still passes unmodified under the new rules. `sharedRoadmaps` was deliberately left untouched (tracked separately by issue #188). See `.claude/rules/auth-security.md`'s "Server-side data caps" entry for the full new-rule list.

### Fixed
- **`changelogDrawer.js` hardcoded the literal string `'Ascent'` instead of using the shared `brand.js` helpers (issue #193).** Root `CLAUDE.md`'s brand rule requires every occurrence of the product name to route through `brand.js` so a future rename or white-labeled variant is a one-file change — this was the one spot in the drawer's kicker label that bypassed it. Now imports `BRAND_NAME` from `brand.js` instead, with no visible change to the rendered label.
- **`deploy.yml` had no concurrency group, so two rapid pushes to `main` could deploy out of order (issue #189).** Two overlapping `deploy` job runs had no ordering guarantee on which `action-hosting-deploy@v0` call finished last — the slower-starting-but-earlier-commit job could finish after the newer-commit job, briefly serving an older commit as "live" right after a newer one was already deployed. Added a `concurrency` block (group keyed on the workflow name + `main`, `cancel-in-progress: false`) so overlapping deploys from rapid successive pushes now queue and run strictly in commit order instead of racing.
- **A missing `FIREBASE_CONFIG` secret only logged a warning instead of failing the deploy job, risking a silent deploy of a fully non-functional app to production (issue #190).** `.github/workflows/deploy.yml`'s "Write Firebase client config" step used to fall back to `firebase.config.example.js`'s placeholder values whenever the `FIREBASE_CONFIG` secret was unset, unlike the missing-`FIREBASE_SERVICE_ACCOUNT` case which already correctly gated the deploy step behind `if: steps.ready.outputs.ok == 'true'`. A scenario where the secret got accidentally cleared or missed during a repo/org secret migration would deploy placeholder Firebase keys — every Auth/Database call fails — straight to the `live` production channel, with only a workflow-log warning and no failed CI status to notice it by. The step now exits non-zero (`exit 1`, plus an `::error::` annotation) instead of writing the placeholder file, so a missing secret produces a loud, visible failed CI check.
- **No CI enforcement that `sw.js`'s `CACHE_VERSION` was bumped when `src/**` changed, risking stale JS/HTML mismatches after a deploy (issue #185).** This app has no build step or content-hashed filenames, so a 1-year `immutable` `Cache-Control` on `/src/**`/CSS plus the service worker's own `cacheFirst` runtime cache meant a returning visitor's browser could keep serving old JS for up to a year against a freshly-deployed `index.html`, with nothing checking that `sw.js`'s `CACHE_VERSION` was actually bumped to invalidate it. Fixed with two changes: `.github/workflows/ci.yml`'s `pr-checklist` job now fails a PR that changes `src/**` without also bumping `CACHE_VERSION` in `sw.js`, and `firebase.json`'s `/src/**`/`**/*.css` `Cache-Control` was shortened from `max-age=31536000, immutable` to `max-age=86400, must-revalidate` as a same-day safety net regardless of the CI check. See `docs/architecture.md`'s Build Log for the full writeup.
- **The AI-import validator had no upper bound on topic/phase/section title length, unlike the manual-entry path (issue #186).** `src/core/roadmap/importValidator.js` only checked that a title was non-empty, so a pasted-or-AI-generated roadmap with a tens-of-thousands-of-characters title passed validation cleanly and got persisted as-is — breaking card/row layouts and costing Firebase storage for no benefit. It now shares `MAX_TITLE_LENGTH` (200) with `roadmapStore.js`'s `addItem()`/`updateItem()`, rejecting an oversized item, phase, or section title with a structured error (e.g. `phases[2].sections[0].title exceeds 200 characters`) matching this validator's existing technical error-message convention.
- **A silent lost-update race could revert a Daily-Todo-linked completion on a roadmap that wasn't currently active and hadn't been visited yet this session (issue #184).** Completing two Daily Todos linked to two different topics inside the same not-yet-cached roadmap within the same network round-trip window used to read the identical stale snapshot and write back the whole `items` map with only one topic's `done` flag flipped — whichever write landed last silently reverted the other topic's completion with no error shown anywhere. Fixed with a targeted, per-item Firebase write (`FirebaseAdapter.updateRoadmapItemFields()`, a multi-path `update()` touching only that one item's own fields) instead of a full read-modify-write `saveRoadmap()` — two concurrent completions for different topics on the same roadmap can no longer clobber each other, since each writes to a completely separate database path. See `.claude/rules/roadmap-store.md` for the full technical writeup.
- **Custom roadmaps could be silently corrupted or destroyed by creating two of them back-to-back — a critical, reported data-loss bug (issue #153).** A lost-update race on `meta.startedTemplateIds`: two overlapping `switchRoadmap()`/`createCustomRoadmap()`/`deleteCustomRoadmap()` calls could each compute the array to persist from the same stale snapshot before either had reassigned it, so whichever call's Firebase write landed last would silently erase the other's id — leaving a fully-created roadmap orphaned from the list of "started" templates, which then re-seeded empty (and got overwritten with nothing) the next time it was opened. Fixed by serializing every meta-array mutation (`startedTemplateIds`/`customRoadmaps`/`favoriteRoadmapIds`) behind a single in-module queue (`roadmapStore.js`'s `serializeMetaMutation()`) so each one always computes from every previously-queued mutation's already-applied result. Three further, related fixes shipped alongside it: `flush()` no longer re-reads the shared `items`/`templatePhases`/`activeTemplateId` after its own network write settles, which could previously stamp a different (newly-switched-to) template's data into the wrong template's local cache slot; a failed save is now automatically retried with exponential backoff (`scheduleSaveRetry()`) instead of failing once and silently giving up forever, and the dashboard's save badge (previously claiming "retrying…" when nothing was ever scheduled to retry) now shows real retry countdown/status plus a manual "Retry now" button; and `onboarding.js` now subscribes to live store updates like every other data-driven page, instead of reading the store exactly once and never re-rendering. `firebase/database.rules.json` also now deploys automatically on every push to `main` (`.github/workflows/deploy.yml`), closing a related gap where the live rules could silently drift from the repo with no CI signal. See `.claude/rules/roadmap-store.md` for the full technical writeup.

### Added
- **Automated knowledge-graph updates after every merge to main.** New `.github/workflows/graph-update.yml` regenerates `graphify-out/` (the committed code-graph knowledge base from issue #210) via `graphify update .` on every push to `main`, and opens a PR with the diff for review rather than pushing directly — keeps the graph AI assistants query from going stale without needing anyone to remember to run it manually.
- **Daily automated Realtime Database backup (issue #130).** A new scheduled GitHub Actions workflow (`.github/workflows/db-backup.yml`) exports the entire Firebase Realtime Database daily, encrypts it (AES-256, this repo is public so plaintext artifacts would leak every user's data), and uploads the ciphertext as a 30-day-retention build artifact, reusing the existing `FIREBASE_SERVICE_ACCOUNT` deploy secret plus a new dedicated `BACKUP_ENCRYPTION_KEY` secret. Previously the only backup path was issue #18's manual, user-initiated, per-user export — there was no operator-side recovery mechanism for an accidental rules deploy, a compromised service account, or a platform incident. ### Security
- **Firebase database rules now enforce, server-side, almost every data cap this app already enforced only in client-side JavaScript (issue #122).** Previously, any authenticated user (guest or real account) could bypass the app entirely and write directly to the Realtime Database REST API against their own uid-scoped paths with none of `src/core/roadmap/limits.js`'s length caps applied — a storage-cost/DoS/data-integrity gap, not a cross-user confidentiality leak (every path was already correctly uid-scoped). `firebase/database.rules.json` now caps: a roadmap item's `title` (≤200), `notes` (≤5000), and each resource's `label` (≤120)/`url` (≤2048); `meta.customRoadmaps`' `title` (≤200)/`description` (≤1000) and array size (indices `0`-`999`, the same index-whitelist idiom issue #177 already established for `favoriteRoadmapIds`, since Realtime Database rules can't call `numChildren()` reliably across CLI versions); `activityLog` keys (must be a real `YYYY-MM-DD`-shaped date) and values (0-5000); and the top-level `reports/{reportId}.screenshotB64` (≤700,000 chars, matching `screenshotCapture.js`'s 500KB cap plus base64 inflation). `roadmapStore.js`'s `createCustomRoadmap()` now clamps (not rejects) an oversized title/description to these same caps, so a legitimate write can never exceed what the new rule allows. A per-roadmap item-count cap remains client-side only (`MAX_ITEMS_PER_ROADMAP`, `addItem()`) — Realtime Database rules genuinely cannot count a map's children, same documented limitation `favoriteRoadmapIds`/`hiddenTemplateIds` already accepted. Feedback-report rate limiting (`feedbackRateLimit.js`) also remains client-side/good-faith only — Realtime Database rules have no way to express "reject if this uid wrote N times in the last M seconds"; closing that specific gap would need a Cloud Function, out of scope here. Every new cap was verified against a real local Firebase emulator (not just assumed to parse) — see the new `tests/e2e/dataCapRules.test.js` rules-emulator suite.

### Fixed
- **A brand-new guest/sign-up could get stuck bouncing between `/onboarding` and `/app` without ever starting a roadmap** (issue #58 follow-up, regression from the multi-roadmap work): "Continue as guest" (and every sign-in path) navigates to `/onboarding` immediately after the Firebase auth call resolves, without waiting for `roadmapStore.setUser()` — which is still resolving its own async Firebase read at that point. `renderOnboarding` could therefore mount using the store's pre-`setUser()` placeholder default (`activeTemplateId = 'java-backend'`), not the fresh user's real state. Picking the Java Backend Engineer card then matched that placeholder and hit `pickTemplate`'s "you're already on this roadmap" shortcut, which just navigated to `/app` without ever calling `switchRoadmap()` — leaving `onboardingDone` still `false`, so the dashboard's own guard immediately bounced back to `/onboarding`. Fixed by only treating a same-id pick as a no-op once onboarding was already done at render time (`isSwitchingTemplate`), so a first-time pick always calls `switchRoadmap()` regardless of the placeholder default. Also fixed a stale assertion in `tests/e2e/onboarding.test.js`'s multi-template regression test that checked a template's "In progress" badge before it had actually stopped being the active one.
- **The previous template's content could silently reappear after switching roadmaps** (issue #51 follow-up, data-integrity bug): reported from real usage — switching to a different starter template would sometimes show the *previous* template's items (including checked-off progress) instead of the freshly-picked one, reproducing on nearly every switch. Root cause: `initFromTemplate()` resets local state synchronously but can't cancel a debounced save from the previous template that's already in flight to Firebase; that save's echo arrived after the switch, was no longer recognized as an echo (`lastFlushedStr` had been reset), and got applied as if it were genuinely new data — overwriting the new template's items with the old template's, both in memory and in `localStorage`. Fixed by tagging every saved roadmap payload with the `templateId` it was written for and having the realtime listener reject any incoming update whose `templateId` doesn't match the currently active template. `firebase/database.rules.json` updated to validate the new field. 4 new regression tests in `tests/integration/roadmapStore.test.js` deterministically replay the exact race (verified to fail without the fix, pass with it) rather than relying on flaky real-time timing.
- **Stale `setUser()` calls could clobber a newer sign-in's state** (issue #51 follow-up): Firebase's `onAuthStateChanged` can fire in quick succession — most notably deleting an account and immediately signing up again with the same email. Because `setUser()` does an async Firebase round-trip before resolving onboarding status, a slower/older call that was still in flight could finish *after* a newer call and overwrite its correct state with stale data, which could manifest as a freshly-recreated account skipping the onboarding picker. `roadmapStore.js` now tracks a `stateCallId` generation counter (shared with `initFromTemplate`) and any call whose id has been superseded by a newer one aborts without mutating state after its `await`s resolve.
- **Onboarding template cards rendered at inconsistent heights within the same row** (issue #51 follow-up): `.template-card` had no `height: 100%` and used `display: grid` with content-sized rows, so a card's box only ever sized to its own text — cards with a longer description or an extra footer button (e.g. the "Start blank" card's "How do I build my own?" link) visibly stuck out below their row-mates instead of matching them. `.template-card` is now a `height: 100%` flex column so it fills the grid's stretched row height, and `.template-card-count` gets `margin-top: auto` so the topic-count footer stays flush to the bottom of every card in a row regardless of description length. `.template-card-hidden` (the "Show hidden templates" restore cards) gets the same treatment. Also fixed a regression introduced while making this change: switching from `display: grid` (which used `justify-self: start`) to `display: flex` briefly caused the "How do I build my own?" button's text to center itself (the browser default for `<button>`), fixed with an explicit `align-self: flex-start; text-align: left`. Verified visually at 390px/820px/1440px viewport widths. See the new "Card/grid layout" convention in `CLAUDE.md`/`AGENTS.md` for the pattern going forward.
